Reflections on Code Reads #12: “Big Ball of Mud”

By "Big ball of mud", the author is referencing an article written by Brian Foote and Joseph Yoder, describing the de facto standard in the professional practice of software engineering: poorly designed, fragile messes. Professional programmers build big balls of mud, because they work. "Not every backyard storage shack needs marble columns", the article references Foot and Yoder. In software engineering, it's better to have something working and on the market, even if it's a mess under the hood. The ball of mud is better than a delayed launch of a product that's achieved architectual perfection.

To programmers destined to encounter a ball of mud, there are three strategies available:

  1. Maintain and evolve (“Keep it Working”)
  2. Throw it out and start over
  3. “surrender to entropy and wallow in the mire”

The big advantage a program running on the market has is that it has users, and feedback. "Maintanence is learning" and "function melts form".

Reflections on Quora post

In his comment, Brian Overland argues professional computer programming can be broken down to three or four separate concerns:

  1. System analysis - analyze the problem domain and define a scope of project requirements.
  2. UI design
  3. Program design - design how the system will be built, structure, data, algorithms and object classes.
  4. Coding - Implementing the specifications by writing code in a programming language.

Brian points out that for smaller projects, all the steps may be performed by a single programmer. In larger, more complex systems the first step in particular is of great importance.

Reflections on DevOps for hybrid cloud: an IBM point of view

Continuous devlivery (CD) shortens the distance between the software development lifecycle and feedback. Feedback can be provided by a program's users, or by analytics microservice; both of which create a feeback loop for the software team.

Cloud architecture, with the use of microservices offer elastic scaling and multi-tenancy. The cloud also allows an increased capacity for automation as well as flexibility in infastructure decisiions and speed in time to market.

In practice, IBM points to :

  • Design thinking
  • lean startup
  • Agile
  • Cloud
  • Security
  • application monitoring
  • analytics

To accomplish these practices, the IBM team promotes a constantly improving work environment; autonomous software teams; transparnecy with metrics that offer learning and improvements, all allowing teams to fail fast in staging environments.

IBM's "Cloud Garage Method":

  • think
  • code
  • deliver
  • run
  • manage
  • learn

The opposite of cloud based software is termed, "system of record". A Hybrid cloud bridges the gap between disparate architecure methods.
