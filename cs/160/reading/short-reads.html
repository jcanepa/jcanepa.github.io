Reflections on Code Reads #12: “Big Ball of Mud”

By "Big ball of mud", the author is referencing an article written by Brian Foote and Joseph Yoder, describing the de facto standard in the professional practice of software engineering: poorly designed, fragile messes. Professional programmers build big balls of mud, because they work. "Not every backyard storage shack needs marble columns", the article references Foot and Yoder. In software engineering, it's better to have something working and on the market, even if it's a mess under the hood. The ball of mud is better than a delayed launch of a product that's achieved architectual perfection.

To programmers destined to encounter a ball of mud, there are three strategies available:

  1. Maintain and evolve (“Keep it Working”)
  2. Throw it out and start over
  3. “Surrender to entropy and wallow in the mire”

The big advantage a program running on the market has is that it has users, and feedback. "Maintanence is learning" and "function melts form".

I call balls of mud "Spaghetti code". In my time as a web application developer, I've had to grab a fork more than once.

Reflections on Quora post

In his comment, Brian Overland argues professional computer programming can be broken down to three or four separate concerns:

  1. System analysis - analyze the problem domain and define a scope of project requirements.
  2. UI design
  3. Program design - design how the system will be built, structure, data, algorithms and object classes.
  4. Coding - Implementing the specifications by writing code in a programming language.

Brian points out that for smaller projects, all the steps may be performed by a single programmer. In larger, more complex systems the first step in particular is of great importance.

Reflections on DevOps for hybrid cloud: an IBM point of view

Continuous devlivery (CD) shortens the distance between the software development lifecycle and feedback. Feedback can be provided by a program's users, or by analytics microservice; both of which create a feeback loop for the software team.

Cloud architecture, with the use of microservices offer elastic scaling and multi-tenancy. The cloud also allows an increased capacity for automation as well as flexibility in infastructure decisiions and speed in time to market.

In practice, IBM points to :

  • Design thinking
  • lean startup
  • Agile
  • Cloud
  • Security
  • application monitoring
  • analytics

To accomplish these practices, the IBM team promotes a constantly improving work environment; autonomous software teams; transparnecy with metrics that offer learning and improvements, all allowing teams to fail fast in staging environments.

IBM's "Cloud Garage Method":

  • think
  • code
  • deliver
  • run
  • manage
  • learn

The opposite of cloud based software is termed, "system of record". A Hybrid cloud bridges the gap between disparate architecure methods.

Takeaways from The Ineviatble Pain of Software Development

The paper's official title goes on to include, Including of Extreme Programming, Caused by Requirements Volatility. In his opening paragraph the author sets out to examine project management methodologies in software projects, specifically extreme programming when set against project requirements. He later states the paper's aim - to answer the question of why there will never be a silver bullet in software engineering.

Next, the author describes the process of building his first software project. He describes his workflow as "seat-of-the-pants build-it-and-fix-it-until-it-works method", where he discovered some requirements, coded a little, discovered a little more, coded a little more, and so on. The subsequent steps tackle testing, which leads to discovery of new requirements and bug squashing. "I was discovering requirement flaws and correcting them during coding as a result of what I learned during coding".

After a long career in software programming, he "still feels like skiing with an avalanche following closely behind." He blames the seldom-aknowledged elephant in the room: ever-changing requirements.

There is no silver bullet in software engineering! Issues arise from either the "essense" or the "accidents". The accidents are everything that's used to provide a software solution (programming language, tools and methods). The essense is the the project's requirements, "deciding precisely what to build", which Brooks calls the most important part, and the one hardest to fix down the line.

The author reveals the paper's main argument, that the central problem in software engineering is problems of essence. He, like Brooks belives there will never be so-called "silver bullet" in software engineering. Barry goes on to explain that since 1968, software engineering methodologies have improved parts of the "accident" space of software engineering. However, none of these popular methodologies have been equipt to sufficiently address issues that arise from the essence (usually a change of a project's requirements during development).

The reason the essence space is so difficult to capture, is because requirements change and they are misunderstood. A study cited in the paper states of all application maintainence, 20% is correcting errors and 80% is changing requirements. The Belady-Lehman graph shows the fragility of a legacy codebase as a function over time.

The paper goes on to layout the most popular software engineering methodologies: Waterfall, Structured programming and the Bild-and-fix model.

Summary of why Software Is Hard

Wilson's post on software starts with a quote from Dreaming in Code, "Software is hard". The autor attempts to answer the question posed in the book, "Why is software a never-ending state of crisis?...Why can't we bild software the same way we build bridges?".

The author does a quick summary of the books details before providing insight on some of the fundamental issues at OSAF. He hilights the problem of CEO hubris and of having no pressure of (time or money) constraints to ship.

He goes on to explicitly list out the Chandler project's problems:

  1. Chandler's peer-to-peer requirement was trivial. It's inclusion as a requirement handycapped the project unnecessarily.
  2. Building software on multiple platforms from the start
  3. Language selection and hiring of non-fluent programmers to work in the language
  4. Ambiguous, fluid project definition and unrestrained scope creep. The democratic decision-making style of the company compounded both of these issues. Decisions to "invent something entirely new from first principles."
  5. No decision is final. Years are spent revisiting and re-thinking complex concepts that have already been decided on.

There's some cost point, some number of development man-months, at which each of those features becomes simply not worth the expense. The problem is that we're never quite sure what a feature is going to cost until it's done.

Kyle Wilson Software Is Hard

The author shows one metric for measuring programming: lines of code. At the same time he cautions, "There's not really any good way to measure the complexity or scale of a piece of software. . .Lines of code is a lousy metric, but it's the only one we've got." That said, it's impossible to discern the magnitude of programmers by looking at a finished product. Analysing complexity of completed code isn't any better.

The idea of measuring complexity where n is number of lines of code as:

  • O(n) - a well designed program where lines of code are tightly coupled to their limited scope.
  • O(n^2) - a poorly designed program where any line of code can call any other line.

This difference in orders of complexity illustrates the importance of object oriented programming and a well-designed system, and cautions on the use of procedural style coding.

"the nature of software is that the problems are always different"
